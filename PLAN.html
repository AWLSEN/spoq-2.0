02 / TIERED ACCESSAnonymous → Identified → Paid. The funnel replaces the login wall.

Every user walks through three doors in order. The first two are free and invisible. The third is where money changes hands — and only if the user wants to reach it. This replaces the "sign in first" pattern entirely.

TIER 00  ·  ANONYMOUS
Just type
No login. No card. No account.
  • Landing → type → agent replies
  • GLM-powered info tasks (research, recipes, explanations)
  • Read-only browser (view pages, no logins)
  • 3 tasks / IP / day, 5-min sandbox TTL
  • Chat history lives in sessionStorage only
  • No card, phone, email, persistence
TIER 01  ·  IDENTIFIED
Sign in — only when needed
Free. Triggered by the agent, not by us.
  • Persistent sandbox (wakes in 0.9s forever)
  • Chat history saved
  • Phone, email provisioned on request
  • Agent can log in to sites on user's behalf
  • Daily usage cap (higher than anon)
  • No spending yet (no card)
TIER 02  ·  PAID
When the agent spends
$15/mo Plus  ·  $49/mo Developer
  • Virtual card issued (Stripe)
  • Claude / GPT upgrade path for hard tasks
  • Unlimited tasks
  • Priority compute / bigger sandbox
  • Pro Mode toggle (harness picker, BYO key, SSH)

Key insight: The login gate is triggered by the agent when it needs something, not by us at the front door. "I need to save our progress →" (tier 01 prompt). "To book this I need a card →" (tier 02 prompt). Grandma never sees a login screen she doesn't understand the reason for.

Six design laws

non-negotiable
LAW 01
No jargon, ever.
"Harness," "agent," "API," "LLM," "MCP," "sandbox," "tool," "model" — none on the default surface. Pro mode may use them. Main UI cannot.
LAW 02
Every button is a verb.
"Book now," "Add card," "Send message." Never "Continue," "Submit," "Confirm," "OK." A verb plus a thing.
LAW 03
The agent talks like a person.
First person, no status codes, no JSON, no backticks. Even when Claude Code is running, the user sees plain English.
LAW 04
Every error is a sentence an apology could follow.
"I couldn't find flights on that date — try another?" Not "500 Internal Server Error." Never show a stack trace or error code.
LAW 05
Resources appear only when needed.
Phone, email, card, login — never up-front. Only when the agent needs one to make progress. One tap to approve.
LAW 06
The grandma test is binding.
Before any screen ships: could a 60-year-old explain to a neighbour what this screen does, using only what's visible? If not, rebuild.

07 / MONETIZATIONThree tiers. Users pay us. They never see a provider bill.

Free tier runs on GLM (cheap). Plus tier is priced to subsidize GLM generously and route hard tasks to Claude. Developer tier is for the 1% who want the raw controls — priced higher, unlocks everything.

FREE
GLM-powered, login-gated actions
Anonymous: 3 tasks/IP/day, info-only.

Identified (Google sign-in): unlimited info tasks, persistent sandbox, phone/email on request, no card.

Subsidized by: GLM's low per-token cost. Break-even even on heavy free users.
PLUS  ·  $15/mo
When the agent spends money
Virtual card issued via Stripe. Automatic Claude/GPT upgrade for complex agentic tasks. Unlimited task volume. Priority sandbox. This is the mainstream paid tier — pitched as "your AI assistant with a credit card."
DEVELOPER  ·  $49/mo
Pro mode switch
Everything in Plus, plus: harness picker (Claude Code / Codex / Gemini CLI / Aider), BYO API key (save ~40% on inference), SSH to sandbox, custom MCP server uploads. Off by default, never affects default UX.
WHY USERS PAY US, NOT OPENAI
One card, one bill, one thing that works
Grandma doesn't want to manage API keys across 4 providers. She wants to pay one service and have things happen. SPOQ absorbs the mess: we negotiate volume rates with Zhipu / Anthropic / Stripe / Twilio / Resend and charge a flat monthly rate on top.

08 / VERIFICATION LOOPSEvery phase has a gate. Every task has a judge. Every regression alerts.

"Strong verification loop" means: we don't ship a phase without passing a test, we don't trust the agent without a judge, and we don't deploy without a canonical task suite running against prod. Four loops, all automated, all visible.

Loop 01  ·  per-phase gate

When: end of each roadmap phase
Phase doesn't ship without a demo-able outcome + a smoke test.

Phase 0 gates the entire project (GLM + Claude Code spike). Each later phase has a specific assertion: "sandbox spawns in <2s," "agent replies to 10/10 canonical tasks," "JIT resource UX is one-tap." Fail = rebuild.

Loop 02  ·  canonical task suite

When: every 6 hours against production
20 real grandma tasks run continuously. Success rate ≥ 85% or alert.

GitHub Actions cron spawns a test account, runs all 20 tasks sequentially, records reply latency + tool-use success + task completion. Results posted to a dashboard and a Discord channel.

Loop 03  ·  LLM judge

When: every completed run, test + live
A Claude Opus judge grades every task: did the agent actually do what was asked?

Judge sees the full transcript + final artifact (receipt, file, URL). Binary grade per task, with a written rationale. Cheap per-judgment (~$0.02). Catches subtle regressions that pass/fail tests miss.

Loop 04  ·  grandma review

When: before any UI change ships
A non-technical person tries the screen cold. If they fumble, the screen rebuilds.

Sam's dad (60yo, Delhi) is the canonical tester. Each UI change is recorded as a short Loom of him using it. Friction = feedback, shipped with the PR.
